2026-07-23 · 代号5 · 接入 Binance USDT 批量代付 + 聚合对账第二范式(多对多)

内容:代号5 原有 5 平台都是「订单号 1:1 追加」(enrich_admin_columnar)。Binance 批量代付没有订单号、一个收款 ID 对应多笔打款,属多对多聚合对账,与逐行追加不同构,故新增第二种范式并一次性写通用引擎:platform_engine 加 derive_series(按列取值,支持去前缀 BIN- / 正则 USDT\s*([\d.]+))、aggregate_by_keys(按「收款ID + 日期」groupby 求和+计数)、reconcile_aggregate(两侧 full-outer,四状态:一致/金额不符/平台缺失/平台多余,支持金额容差)。platform_spec.PlatformSpec 加可选 recon_mode(None=逐行 / aggregate=聚合)与 recon(平台/admin 取值、日期口径、状态标签、输出声明);缺省 None 故对现有 5 平台与代号6 零影响。platform_handlers_5 加通用 AggregateReconHandler(read 读平台模板 + build_reconciliation 全程由 spec.recon 驱动、无平台专属分支),register_handler("aggregate_recon", …)。config5 加 BINANCE 常量与 BUILTIN_SPECS_5 声明(priority 60、handler aggregate_recon、recon.date_match_mode="t1_window"),生成 platforms/5/binance.json。app5.main 在列式 enrich 后遍历 recon_mode=="aggregate" 的 spec 调 build_reconciliation 收集 {sheet:df};write_output_5 加可选 extra_sheets,写完既有 4 sheet 后净增;read_admin_5 由写死 xlrd 改按扩展名 excel_engine 自适应(新 admin 为 .xlsx,旧 .xls 路径不变)。

原因:用户要求新平台尽量也用配置文件驱动、免重打包。Binance 平台本身能 100% 纯 JSON——但前提是「聚合对账」这一计算范式(过滤→派生→groupby→full-outer→状态判定→独立 sheet)现有逐行 1:1 引擎不具备,须一次性写入引擎(与当初 enrich_admin_columnar 同理:写一次,之后同型平台纯 JSON)。日期口径经真实同期样本实测:平台常在 admin 订单次日(T+1)打款,严格按天会把完美配对误判成大量虚假缺失/多余,故用户选定 t1_window(平台打款日 pd 归属:当天有 admin 归 pd,否则前一日 pd-1 有 admin 归 pd-1,都没有留 pd=平台多余);口径为 JSON 字段可切 exact/period/t1_window。

效果:Binance 100% 纯 platforms/5/binance.json 声明驱动、免重打包;未来同为聚合型的新平台只需再丢一个 JSON、零改仓库。输出 data/output/代付对账结果_{YYYYMMDD}.xlsx 在发现 Binance 源文件时净增 Binance-USDT对账 sheet(日期/收款ID/admin应付USDT/admin笔数/平台实付USDT/平台笔数/差额/对账状态),原 4 sheet 逐字节不变(金标准对拍通过);tests/test_platform_plugin_5.py 扩至 15 用例(derive_series 去前缀/正则、四状态判定、三日期口径、Binance handler 端到端、内置键表含 BINANCE、防漂移)。诚实边界:若某聚合平台派生超出「列 + 正则 + 去前缀」(如查汇率表换算、跨表链式 join)仍须插件或改引擎。

2026-07-22 · 代号6 · 平台配置外置化(两层插件,接新平台免重打包)

内容:把"平台是什么、列名叫什么、状态怎么映射、按什么优先级/方向匹配"从编译进二进制的 Python 常量+函数,外移成 exe 旁可编辑的松散文件,分两层:第一层 platforms/6/*.json 声明式(结构类似的平台加一个 JSON 即接入),第二层 platforms/plugins/*.py 插件(需自定义清洗的疑难平台,注册自定义 handler)。新增三个与代号无关的模块:platform_spec.py(PlatformSpec/DirectionSpec/OutputSchema、规范内部列 CANON、handler 注册表)、platform_engine.py(通用读取/归一化查找表/按优先级 enrich/平台多余行,并集中低层读取原语)、platform_loader.py(定位 platforms/ 目录、内置→JSON→插件深合并、隔离导入与失败隔离)。config6.py 新增 BUILTIN_SPECS_6(用现有列名常量拼装的内置真值);app6.py 的 read_*/build_*_lookup/enrich_admin_6/scan/main 改为薄壳委派引擎,保留 build_betcat_lookup_6/build_goldenpay_lookup_6/enrich_admin_6 三处旧签名。Goldenpay 收/付列名差异改由 directions.<dir>.columns 声明,原命令式 df.rename 消失。打包 bank_integration(.mac).spec 构建后脚本增拷 platforms/ 到 exe 旁。

原因:此前平台接入逻辑与配置全部写死在 .py 被 PyInstaller 编进二进制,每加一个三方平台都要改源码→重跑打包→重发 126M 包。而 onedir 包本就把 template//data/ 放 exe 旁松散文件、get_project_root() 已按 frozen 用 exe 目录做根,且 onedir 能运行时 importlib 加载 exe 旁松散 .py(依赖 pandas/openpyxl/xlrd 已在包内)——具备外置化的全部前提,只差一个"运行时从外部文件加载平台"的入口。

效果:结构类似的新平台加一个 JSON、疑难平台加一个 .py,均无需重打包;唯一例外是新平台需要未打进包的新第三方库(少见,可用 platforms/vendor/ 放纯 Python 依赖规避)。无外置目录/空目录时注册表=内置三平台,行为与外置化前逐字节一致。用 2026-04 真实样本金标准对拍改造前后产物,6 个 sheet(代收 27248 / 代付 11892 行 + 汇总)逐单元一致;现有 4 个代号6 单测零改动通过,新增 tests/test_platform_plugin_6.py(内置回退、JSON 覆盖/新增、插件注册与失败隔离、方向列覆盖、内部列不泄漏、防漂移)11 用例全通过。
