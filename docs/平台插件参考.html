平台插件参考(接新平台免重打包)

代号6 的平台定义已外置到 exe/仓库根旁的 platforms/ 目录。改这里的文件后重新运行程序即可生效,无需重新打包。本文档是 JSON schema 与插件契约的唯一权威说明;面向使用者的简版见 platforms/README.md。

1. 目录与加载顺序

platforms/
├── 6/*.json         代号6 声明式平台（每平台一个）
├── plugins/*.py     疑难平台插件（自定义 handler）
└── vendor/          可选：纯 Python 第三方依赖（自动加入 sys.path）

加载器 platform_loader.load_platform_registry("6") 依次:内置默认(config6.BUILTIN_SPECS_6)→ JSON 深合并 → 插件覆盖/新增,最后按 priority 升序返回。同 key 深合并(JSON/插件只需写增量);enabled:false 停用(含内置平台);任一 JSON/插件出错只打中文 warning 跳过,不影响其它。目录不存在/为空时注册表=内置三平台,与外置化前逐字节一致。

定位规则:环境变量 BANK_PLATFORMS_DIR 优先,否则 get_project_root()/platforms(打包后=exe 目录,源码运行=仓库根),与 template/、data/ 同构。

2. JSON schema（第一层：声明式平台）

字段必填说明
key是平台标识(大写),= 结果"来源平台"值 / 状态映射键 / merge 前缀
priority是越小越优先(匹配 & 平台多余行追加顺序)。内置 Betcat=10 / Cashnewpay=20 / Goldenpay=30,新平台建议 ≥40
join_col是平台侧关联键列名(对应 admin.订单号)
handler否默认 "generic";插件平台填其注册的 handler 名
handler_params否传给自定义 handler 的参数
sheet否Excel 优先 sheet(CSV 忽略)
use_first_sheet否未命中 sheet 时是否回退首个,默认 true
columns是规范字段 → 源列名(顶层默认,见下)
status_map否原始状态 → 标准状态(成功/失败/处理中/关闭)精确映射
status_prefix_map否状态含动态后缀时按前缀映射(如 USER_REFUND-<单号>)
directions是collection/payout 各含 prefixes(必填,文件名前缀)、可选 sheet、可选 columns(逐键覆盖顶层)
enabled否false 停用

columns 的规范字段名（固定集合，取自 platform_spec.CANON）

规范字段含义
platform_no平台内部流水号 → 结果"平台流水号"
amount平台金额(命中判断依据,必须能取到)
fee手续费
status原始状态(中文)
status_desc原始状态描述(英文,若有则优先于 status 映射)
finish_time完成时间(结果"交易日期"首选)
create_time创建时间(完成时间为空时回退)

示例：Goldenpay（收/付列名不同，用 directions 覆盖）

{
  "key": "GOLDENPAY", "priority": 30, "join_col": "商户单号",
  "handler": "generic", "use_first_sheet": true,
  "columns": { "fee": "手续费", "status": "订单状态",
               "finish_time": "完成时间", "create_time": "创建时间" },
  "status_map": { "支付成功": "成功", "成功": "成功", "支付失败": "失败", ... },
  "directions": {
    "collection": { "prefixes": ["goldenpay收款"], "sheet": "代收订单导出",
                    "columns": { "platform_no": "订单号",   "amount": "订单金额" } },
    "payout":     { "prefixes": ["goldenpay兑换"], "sheet": "代付订单导出",
                    "columns": { "platform_no": "订单编号", "amount": "金额" } }
  }
}

3. 插件契约（第二层：疑难平台）

当读取/清洗声明式表达不了(两行合并、聚合、多 sheet 联表、倒推汇率、PDF 等)时,写 platforms/plugins/*.py。可选 CODE="6"/CODES=["5","6"] 限定代号。两种写法:

  • 纯声明:模块级 PLATFORM = {...}(同 JSON schema),走内置 generic handler。
  • 自定义 handler:def register(api): 内 api.register_platform(spec_dict, handler=你的handler实例)。

handler 需实现:

def read(self, spec, direction, filepath) -> pandas.DataFrame: ...
def build_lookup(self, spec, direction, df) -> pandas.DataFrame: ...

build_lookup 返回的查找表必须:以 spec.join_col(对应 admin 订单号)为 DataFrame 索引;列名用 CANON 内部列名(__no__/__amt__/__fee__/__status__/__desc__/__ftime__/__ctime__)。引擎据此统一取值、映射状态、生成结果 7 列,内部列不会泄漏到输出。可复用 platform_engine 的 read_source_table/dedup_lookup/build_lookup_from_columns。完整示例见 platforms/plugins/example_platform.py.txt。

4. 唯一仍需重打包的例外

当新平台需要一个未打进包的新第三方库时(如 .xlsb 需 pyxlsb、.parquet 需 pyarrow)才需重打包。缓解:纯 Python 依赖放 platforms/vendor/(加载器自动 prepend sys.path);约定新平台优先导出 CSV/XLSX(pandas/openpyxl/xlrd 已在包内,覆盖绝大多数)。此外,输出表结构/汇总口径/admin 主表读取仍是代码,不在插件覆盖范围。

5. 代号5(代付对账)已外置化

基础设施 platform_spec/engine/loader 与代号无关,load_platform_registry(code) 传不同 code 即可复用。代号5 已推广落地,与代号6 的差异:

  • 用列式引擎 enrich_admin_columnar(不是代号6 的 enrich_admin_generic),按平台原生列名取值(不走 CANON 重命名),支持多 admin 关联键、币种、到账金额、机构、手续费三模式(fee_mode/arrive_mode)、EPIN 机构门槛(admin_match_filter);输出列由 SCHEMA_5.column_plan 声明。
  • 内置 5 平台:SUPERPAY 纯声明(handler:"generic5");IBFYPAY/WANGGUYPAY/EPIN/PHONECARD 挂 platform_handlers_5 的自定义 handler(其 read/build 保留在 app5,由 main 按 BUILTIN_PLATFORM_KEYS_5 分派)。
  • 加结构类似 SUPERPAY 的新直连平台:在 platforms/5/ 放一个 JSON(声明 join_col/admin_join_col/columns/fee_mode/arrive_mode/currency_col|default/status_map/directions.payout.prefixes),main 通过通用 handler 自动扫描+建表+匹配,免打包。疑难平台走 platforms/plugins/*.py(CODES=["5"])。
  • 代号5 仅支持 .xls/.xlsx(不支持 CSV);余额汇总/匹配金额差异对比属 app5 输出形态,不在声明层。

后续 3→1/2 可同法推广(代号1/2 的农行正则/中行日期范围/华美 PDF 等硬编码分支需先降维成声明式选项或 handler)。

6. 代号5 聚合型平台(recon schema,第二种范式)

上面 5 个平台都是「订单号 1:1 追加」。平台没有订单号、一个收款 ID 对多笔的(如 Binance USDT 批量代付)走聚合对账范式:两侧各按「收款ID + 日期」groupby 求和+计数 → full-outer 比对 → 四状态 → 产出独立 sheet(原有 sheet 逐字节不变、净增)。设 recon_mode:"aggregate" + handler:"aggregate_recon",取值/口径/输出全部由 recon 块声明,通用 AggregateReconHandler 无平台专属分支。加同型新平台只丢一个 JSON,免改代码免重打包。

字段含义
recon.date_match_mode日期口径:exact(严格按天)/period(忽略日期整期)/t1_window(T+1 窗口对齐:平台打款日 pd 当天有 admin 归 pd,否则前一日 pd-1 有 admin 归 pd-1,都没有留 pd=平台多余)/t1_shift(严格 T-1:平台日 pd 恒定前移到 pd-1、不试同日,用于平台日=上传日 T 而内容为 T-1 数据的场景,如 Binance)
recon.amount_tolerance金额容差,|差额|≤容差 判「一致」(默认 0)
recon.output_sheet独立对账 sheet 名(如 Binance-USDT对账)
recon.platform平台侧取值:sheet/header_row(表头所在行,0-based)/id_col/amount_col;日期取自 date_from_filename(文件名正则,第1组)或 date_col(列)
recon.adminadmin 侧:filter_col+filter_prefix(前缀筛选)、status_col+status_include(状态白名单)、id_col+id_strip_prefix(去前缀取ID)、amount_col+amount_regex(正则取额,第1组)、date_col
recon.labels四状态标签:consistent/amount_diff/platform_missing/platform_extra
recon.output_columns8 个显示列名(按固定语义序:日期/收款ID/admin金额/admin笔数/平台金额/平台笔数/差额/状态)

派生仅支持「列 + 正则(amount_regex)+ 去前缀(id_strip_prefix)」;超出者(查汇率表换算、跨表链式 join)才需插件或改引擎——外置化的诚实边界。

示例:Binance(聚合型,macro 合并后格式 header=0、逐行 date 列、严格 T-1)

{
  "key": "BINANCE", "priority": 60,
  "handler": "aggregate_recon", "recon_mode": "aggregate",
  "join_col": "Recipient's Account information (Required)",   // 占位(聚合不建查找表)
  "recon": {
    "date_match_mode": "t1_shift", "amount_tolerance": 0,     // 平台日=上传日 T,内容属 T-1
    "output_sheet": "Binance-USDT对账",
    "platform": {
      "sheet": "数据", "header_row": 0,                       // macro 合并输出(找不到 sheet 回退首个)
      "id_col": "Recipient's Account information (Required)",
      "amount_col": "Amount (Required)",
      "date_col": "date"                                       // 合并文件逐行日期列(上传日 T)
    },
    "admin": {
      "filter_col": "其他", "filter_prefix": "BIN-",
      "status_col": "状态", "status_include": ["已完成"],
      "id_col": "其他", "id_strip_prefix": "BIN-",
      "amount_col": "奖品名称", "amount_regex": "USDT\\s*([\\d.]+)",
      "date_col": "日期"
    },
    "labels": { "consistent": "一致", "amount_diff": "金额不符",
                "platform_missing": "平台缺失", "platform_extra": "平台多余" },
    "output_columns": ["日期","收款ID","admin应付USDT","admin笔数",
                       "平台实付USDT","平台笔数","差额","对账状态"]
  },
  "directions": { "payout": { "prefixes": ["usdt奖品发放信息","binance-","merged-"] } }
}

platforms/5/binance.json 须与 config5.BUILTIN_SPECS_5 的 BINANCE 声明一致(测试 test_repo_json_matches_builtin_specs code"5" 守护);recon 非特判合并键,深合并时整块替换,故 JSON 须完整镜像。
